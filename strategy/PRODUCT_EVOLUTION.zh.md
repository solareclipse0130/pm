# 产品演进宏观思路

这份文档记录 MVP 之后的产品演进方向。当前项目已经完成一个可运行、可验收的 Project Management MVP，下一步目标是从“证明能跑”转向“做成一个真正可用、可持续演进的产品”。

## 核心定位

当前产品可以收窄为：

> 一个本地优先、AI 可操作看板的轻量项目管理工具，适合个人开发者和小团队快速规划、拆解、整理任务。

后续所有功能都应围绕这个定位判断是否值得加入。

在添加功能前，优先回答：

- 这个功能是否让用户更容易管理项目？
- 这个功能是否强化 AI 辅助项目管理的差异点？
- 这个功能是否仍然保持工具轻量、直接、可理解？
- 这个功能是否会让当前代码和数据模型过早复杂化？

## 第一阶段：从 MVP 到日常可用

MVP 已经具备登录、看板、持久化、AI 创建/编辑/移动卡片。下一步最值得补的是基础产品能力：

- 卡片字段：优先级、截止日期、标签、负责人。
- 搜索和过滤：按关键词、状态、标签、优先级过滤。
- 活动记录：记录什么时候改了什么。
- 操作确认或撤销：尤其是 AI 大规模修改看板前。
- 更完整的空状态、错误状态、加载状态。
- 数据导入导出：JSON 或 CSV 备份。

这类功能能让应用从 demo 变成真正能日常使用的工具。

## 第二阶段：强化 AI 产品价值

AI 不应只停留在“修改 board JSON”。更有价值的方向是让它成为项目助理：

- 把散乱想法拆成任务。
- 根据当前看板总结项目进度。
- 找出阻塞项和风险。
- 规划下一周工作。
- 把一张大卡片拆成更小任务。
- 根据完成情况生成周报。
- 审查当前看板是否合理。

更重要的是，AI 操作应逐步从“直接修改”升级为：

```text
用户提出需求
-> AI 给出修改计划
-> 用户确认
-> 系统执行修改
-> 记录操作结果
```

这样可以提高可靠性，也更适合真实产品。

## 第三阶段：真实用户系统

MVP 当前使用硬编码登录，仅适合本地测试。产品化时需要补真实认证边界：

- 后端登录接口。
- 密码哈希或外部身份提供商。
- session 或 token。
- 用户隔离。
- 每个用户自己的 board。
- 后续再考虑团队、权限和协作。

这一步是从本地工具走向真实产品的重要分界线。

## 第四阶段：数据模型逐步结构化

当前把 board 存为 JSON 很适合 MVP。随着功能增加，可以逐步拆分数据模型：

- `users`
- `boards`
- `columns`
- `cards`
- `labels`
- `activity_logs`
- `ai_sessions` 或 `ai_messages`

不要一口气重构。等搜索、过滤、活动记录、多用户等功能真正需要时，再自然拆分。

## 建议版本路线

建议按版本推进，而不是无限连续开发。

```text
v0.1 MVP
已完成。登录、Kanban、持久化、AI 侧边栏、Docker 本地运行。

v0.2 Product Basics
补齐卡片字段、搜索过滤、导入导出、基础 UX 改进。

v0.3 Real Auth
加入真实登录、多用户隔离、后端认证边界。

v0.4 AI Workflow
让 AI 从直接修改升级为建议、确认、执行、总结。

v0.5 Team Lite
支持简单团队、共享 board、活动记录和基础权限。

v1.0 Stable Product
稳定部署、完整验收、文档齐备、可长期使用。
```

## 技术原则

后续技术演进应保持“简单但不再临时”：

- 保留 FastAPI + NextJS + SQLite，直到真实需求要求替换。
- 使用 Pydantic 明确 API schema。
- 引入数据库 migration。
- 给 AI 操作加审计、确认和冲突处理。
- 加 CI，至少运行后端测试、前端 lint、单元测试和 e2e。
- 部署前再考虑 Postgres、HTTPS、域名、云服务。

## 判断标准

每次准备添加功能时，优先问：

- 它是否属于当前版本目标？
- 它是否能被清晰验收？
- 它是否有对应测试或手动验收步骤？
- 它是否会影响已有 MVP 能力？
- 它是否值得现在做，而不是放到后续版本？

产品演进的重点不是加得快，而是每一步都能保持清楚、可测、可回退。
